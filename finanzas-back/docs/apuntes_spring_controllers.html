Apuntes · Controllers y Mappings

Spring Boot — cuándo usar GET, POST, PUT y DELETE · ejemplos con tu app de finanzas

1. Anotaciones base del controller

Estas van una sola vez, a nivel de clase:

@RestControllerMarca la clase como controlador REST. Los métodos devuelven JSON directamente (combina @Controller + @ResponseBody).
@RequestMapping("/api/gastos")Ruta base común a todos los endpoints de la clase. Cada método cuelga de aquí.
@RequiredArgsConstructorDe Lombok. Inyecta el service por constructor (la forma recomendada, mejor que @Autowired en campos).
@RestController
@RequestMapping("/api/gastos")
@RequiredArgsConstructor
public class GastoRecurrenteController {
    private final GastoRecurrenteService service;
    // ... endpoints ...
}

2. Los 4 verbos — pulsa para desplegar

Regla mental rápida: GET lee, POST crea, PUT actualiza, DELETE borra.

3. Tabla comparativa

@GetMappingGETLeer / consultarNoSí200 OK
@PostMappingPOSTCrear nuevoSíNo201 Created
@PutMappingPUTActualizar (reemplaza)SíSí200 OK

Idempotente = repetir la misma petición varias veces da el mismo resultado que hacerla una vez. Borrar el gasto 5 diez veces deja el mismo estado (idempotente); crear un gasto con POST diez veces crea diez gastos (no idempotente).
PUT vs PATCH: PUT reemplaza el recurso entero (mandas todos los campos). PATCH actualiza solo algunos campos. Si dudas, PUT es lo habitual para empezar.

4. Anotaciones de parámetros

@PathVariableCaptura un valor de la URL. En /gastos/5 → el 5. Se usa con GET, PUT, DELETE.
@RequestBodyConvierte el JSON del cuerpo en un objeto Java. Se usa con POST y PUT.
@RequestParamParámetros de query: ?mes=4&anio=2025. Para filtros y búsquedas.
@ValidDispara la validación del objeto recibido (@NotNull, @Size...). Junto a @RequestBody.
